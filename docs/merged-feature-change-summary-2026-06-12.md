# 功能改造汇总（截至 2026-06-12）

本文把已合入历史，以及当前工作区里的覆盖率增量，一起按功能模块重新整理，只讲两件事：

1. 改了哪些功能
2. 这些功能现在按什么逻辑工作

不展开文件行号，也不按提交逐条罗列，适合作为产品、交接和回顾用的总览。

## 1. API Key、套餐、资源包与自动切换

这是本仓库改造最多的一块，核心目标是把「一个用户有多个 Key、多种套餐、多种资源包」这件事跑顺。

### 1.1 API Key 校验链路从单点校验变成完整校验

现在 API Key 不是只做“哈希是否匹配”这一层，而是把整条限制链路都纳入校验：

- Key 是否存在、是否启用、是否过期
- 客户端限制、权限限制、模型限制
- 多窗口请求数/费用限制
- 每日费用、总费用、Claude 周费用等配额限制

这样做的结果是：当系统尝试切换到备用 Key 时，备用 Key 也会走同样的完整校验，不会切到一个“虽然存在，但其实已经超限/禁用/过期”的 Key。

### 1.2 个人版多 Key 自动切换

个人版通过 `externalUid` 把同一个用户名下的多个 Key 关联起来。请求失败或当前 Key 不合适时，系统会从这个用户的候选 Key 集合里继续找可用 Key。

当前逻辑重点如下：

- 同一个 `externalUid` 下的 Key 会建立独立索引，方便快速找备用 Key
- 当前 Key 校验失败时，会在候选集合里逐个检查其他 Key
- 备用 Key 同样要通过完整限制检查，才能真正被切过去
- 整个过程在服务端完成，对客户端透明

### 1.3 个人版切换优先级

个人版切换不是“随机换一个”，而是按资源类型排序：

1. `pack-free-3` 这类 AI 修复专属免费包
2. 订阅套餐 Key
3. 普通 Key
4. 资源包 Key

这个优先级背后的业务逻辑是：

- 主进程的 AI 修复请求，优先消耗 AI 修复专属免费包
- 如果用户名下有可用订阅套餐，普通请求优先切到订阅套餐，尽量先吃长期订阅额度
- 订阅和普通 Key 都不满足时，才考虑资源包

### 1.4 `pack-free-3` 免费资源包的特殊逻辑

`pack-free-3` 不是普通资源包，它被当成 AI 修复专属包处理：

- 只有主进程的 AI 修复请求，会把它放到最高优先级
- 非主进程请求不会优先抢占它，只把它当普通候选 Key 看待
- 如果系统本来想优先切到 `pack-free-3`，但实际上没有可用的 `pack-free-3`，会回退到原始 Key 或继续走普通切换链路，而不是直接失败

这套逻辑的目标，是把免费额度尽量留给真正的 AI 修复场景使用。

### 1.5 资源包切换需要 `pack_consent`

资源包不是默认无条件接管的。

当前规则是：

- 如果当前 Key 已经是资源包，可以直接切到其他资源包
- 如果当前 Key 不是资源包，只有带了 `pack_consent` 的 Key，才允许在套餐或普通 Key 不够用时继续切到资源包
- 也就是说，“从套餐切到资源包”必须是显式授权行为

这样做是为了避免用户本来只想用订阅套餐，却被系统静默消耗掉资源包额度。

### 1.6 企业版共享 Key 与企业资源包

企业版和个人版的切换链路已经彻底隔离。

企业版现在通过两个请求头进入独立逻辑：

- `uni_agent_subscription_type=enterprise`
- `uni_agent_subscription_user_id`

系统会根据成员 uid 去企业成员索引里找候选 Key，只在企业 Key 集合内切换，不会混入个人 Key。

企业版的切换规则是：

1. 企业订阅
2. 企业普通 Key
3. 企业资源包

并且企业模式下：

- 不再依赖 `pack_consent`
- 只要该成员在 `memberUids` 里，就视为有权使用这个企业 Key
- 企业配额按 Key 维度整包共享，成员共同消耗

在企业版基础上，又继续扩展了企业资源包、企业提示语、个人请求过滤企业 Key、`packMode` 返回值等规则，让企业和个人的使用边界更清楚。

### 1.7 多窗口限额与窗口重置

API Key 限额从单一限制扩展成了多窗口规则数组，也就是一个 Key 可以同时挂多条限制，例如：

- 5 小时请求数限制
- 5 小时费用限制
- 7 天费用限制

当前逻辑是：

- Key 创建成功后，窗口计数器会立即初始化
- 更新 Key 时，只有窗口结构真的变了，才重置窗口
- 只改额度、不改窗口结构时，不会粗暴清空现有窗口
- 窗口里的费用消耗，按倍率后的真实计费金额累计

另外还补了 `rest_window` 这类控制参数，便于手动重置窗口限制。

### 1.8 过期、禁用与激活模式

API Key 的有效期除了固定过期时间，还支持“首次使用后激活”的激活模式。

围绕这块又做了几类补强：

- 过期或禁用的 Key，本身不能继续用，但仍然可以触发去找同用户的其他可用 Key
- 如果把过期时间改到未来，Key 可以自动恢复激活状态
- 相关错误提示改成更贴近套餐/资源包语义的中文提示

### 1.9 账户绑定与服务倍率

API Key 不再只绑定单一 Claude 账号，而是扩展成支持多平台绑定和倍率控制：

- 支持平台账号或账号组绑定
- 支持平台级服务倍率
- 支持按平台分别控制请求走哪类账号

这让一个 API Key 可以在计费、账户池、资源隔离上做更精细的编排。

## 2. Partner API 与企业版 Partner API

Partner API 已经从单一创建接口，扩展成一整套可运营的外部接入能力。

### 2.1 Partner API 的总体逻辑

Partner API 采用 SHA-256 验签，整体逻辑是：

- 请求参数按 key 排序
- 对对象和数组做稳定序列化
- 统一拼接后追加密钥
- 计算 SHA-256 并转成大写

这样做的目的，是让第三方系统可以稳定创建、更新、查询 API Key，而不用依赖管理后台人工操作。

### 2.2 已扩展出的核心接口能力

Partner API 目前已经覆盖：

- 创建 API Key
- 更新单个 API Key
- 批量更新多个 API Key
- 更新过期时间或激活时间
- 查询用量汇总
- 查询近 30 天用量明细

这些接口都围绕“批量接入、批量运维、批量计费”设计，不再只是简单地发一个 Key。

### 2.3 Partner API 可以控制的业务字段

Partner API 现在支持的关键业务字段包括：

- 各平台账号绑定
- 账号组绑定
- 各平台倍率
- 多窗口 `rateLimits`
- `user_id` / `externalUid`
- `pack_consent`
- 固定过期 / 激活过期
- 企业版成员列表

也就是说，外部系统创建 Key 时，就能把“绑定谁、怎么计费、有什么窗口限制、是否允许切资源包、属于谁、是否企业共享”一次性配好。

### 2.4 企业版 Partner API

在个人版 Partner API 之外，又新增了企业版接口，主要解决“一个企业 Key 需要授权给多个成员共用”的问题。

企业版的核心逻辑是：

- `externalUid` 仍然表示购买者或归属者
- `memberUids` 表示当前有权使用该 Key 的成员列表
- 成员列表变化时，系统会自动维护成员到企业 Key 的反向索引
- 企业成员接口采用“全量覆盖”的方式设置成员，避免局部增删造成状态不一致

这样一来，企业管理员可以用 Partner API 批量创建企业 Key，再统一下发成员授权。

## 3. 多平台账号、调度与转发能力

仓库已经从“以 Claude/OpenAI 为主”扩展成“多平台统一中转”。

### 3.1 DeepSeek 平台

DeepSeek 这一块已经补齐了整套链路：

- 账号存储与加密
- 调度器
- OpenAI 兼容接口
- Anthropic 兼容接口
- 后台 CRUD、测试、启停、重置限流/状态
- 统计和定价接入

也就是说，DeepSeek 已经不是单点转发，而是和 Claude/OpenAI 一样，具备完整的后台管理与计费支撑。

### 3.2 Kimi、GLM、MiniMax 三个平台

后续又继续接入了：

- Kimi
- GLM
- MiniMax

这三个平台都按统一模式补齐了：

- 账号实体
- 加密存储
- 调度器
- Relay 转发服务
- 管理后台接口
- 账号测试与状态恢复

同时补了 Anthropic 协议兼容，因此这些平台不只是走 OpenAI 格式，也能兼容另一套消息协议。

### 3.3 账户级模型映射

四个平台又继续增加了一个非常关键的能力：模型映射。

逻辑是：

- 请求里传入的是“业务模型名”
- 账户上可以配置“业务模型名 -> 平台实际模型名”的映射
- 映射优先走精确匹配
- 精确没命中，再走通配符匹配
- 都没命中，就原样透传

这个能力的价值在于：

- 前端或调用方不用改统一模型名
- 不同平台账户可以各自映射到最合适的真实模型
- 切账号、换平台时，不必同时改调用方的 model 参数

## 4. 计费、成本价与统计口径

计费系统已经从单平台简单计费，扩展成“多平台、多口径、多特殊规则”的统一计费体系。

### 4.1 多平台定价来源

当前定价逻辑大致分成三层：

- 优先使用抓取到的官方价格
- 抓不到时使用镜像价格
- 仍然缺失时再用内置 fallback 价格

这样能保证：

- 价格尽量跟官方保持同步
- 即便上游价格接口异常，系统也不会完全失去计费能力

### 4.2 DeepSeek 计费规则

DeepSeek 这块做了几类关键补强：

- 官方价格抓取
- cache hit / cache miss 成本支持
- 纯 OpenAI-compatible 响应里没有缓存字段时也能兼容计费
- DeepSeek 的费用在统计页里单独归类展示
- DeepSeek Pro 折扣价规则保留

也就是说，DeepSeek 不再只是“能转发”，而是能比较稳定地算出真实费用。

### 4.3 GLM 计费规则

GLM 这块的逻辑更特殊，主要做了两层：

- 国内人民币价格统一按 `/7` 换算成 USD
- 不同请求大小走不同成本价

也就是请求上下文越大，成本单价可能越高，而不是一口价处理。

### 4.4 MiniMax 计费规则

MiniMax 这块补了：

- 价格抓取
- fallback 定价
- M3 512K 模型的分层计费

现在 MiniMax M3 512K 会按上下文窗口大小分档计费，而不是简单按一个统一价格算。

### 4.5 Claude / 通用计费修复

除了新增平台，还修了不少基础计费问题，例如：

- Claude 4.6 系列费用漏记
- 缓存计费不准
- 倍率消耗计算错误
- 窗口限制使用的费用口径与最终计费口径不一致

现在整体方向是：窗口限额、用量统计、费用展示，尽量共用同一套计费口径。

## 5. OpenAI Codex 账号使用量自动保护

OpenAI 账号增加了一套自动停调与自动恢复机制，避免单个账号把大窗口额度打穿。

### 5.1 三种自动保护规则

当前支持三种保护：

1. 5 小时窗口使用量达到 95% 自动停调
2. 周窗口使用量达到 95% 自动停调
3. 周额度按日均摊超限时自动停调

第三种不是简单均分，而是按前几天高、后几天低的递减预算控制，用来限制周额度在前期被一次性打完。

### 5.2 恢复逻辑

停调后不是人工手动恢复，而是自动判断恢复时机：

- 5 小时保护：等 primary reset 到点后恢复
- 周限额保护：等 weekly reset 到点后恢复
- 日均摊超限：到次日本地时区零点恢复

### 5.3 这套保护又补过几次细节修复

后续又修了几类关键问题：

- 使用量停调状态不能被普通限流恢复逻辑绕过
- 管理员关闭保护开关后，已经停调的账号要立即解除停调
- `timezoneOffset=0` 时不能误判为默认时区
- 部分“周限额自动停止调度不生效”的情况被补齐

这套逻辑现在已经不仅是“达到阈值就停”，还强调“停得准、恢复得准、人工干预也能及时生效”。

## 6. 请求明细、日志、追踪与思考链

这一块的目标，是把“请求发生了什么”记录得更完整，同时不把日志搞爆。

### 6.1 请求追踪与日志字段扩展

现在请求链路里补充了很多可追踪字段，例如：

- `sessionId`
- `rawSessionId`
- `userInput`
- `assistantContent`
- `user_id`
- `tools`
- `processType`
- `projectType`

这样后台在看用量明细或问题排查时，不再只知道“花了多少钱”，而是能知道“是谁、在哪类进程、带了什么输入、产生了什么回复”。

### 6.2 请求日志的裁剪和脱敏

为了避免日志过大，又做了很多保护：

- 图片和大段 base64 不再直接落日志
- 请求体会做字段级过滤
- 可以控制是否打印 chunk 级别日志
- 请求完成后再写入 token 用量，减少半途中间态不准的问题

这套逻辑是在“记录更完整”和“日志别炸掉”之间做平衡。

### 6.3 流式与错误场景的补强

后续还补了不少跟稳定性相关的细节：

- SSE 响应不完整时，主动给客户端返回可重试错误
- DeepSeek 请求的 `rawSessionId` 修正
- `x-request-id` 优先透传
- `server_is_overloaded` 等上游异常额外拦截
- 错误日志采集继续加深
- 请求里使用了哪个账号，也会更清楚地记录出来

### 6.4 思考链与翻译

仓库还增加了 reasoning / thinking 的采集与翻译能力：

- 思考链内容可以存储
- OpenAI 风格流里可以把翻译后的 reasoning 内容再注入回去
- 翻译使用增量处理，不必等整段完成
- 翻译 token 也会并入 usage 统计

这个能力主要是为了让前端既能看原始推理流，也能更自然地展示中文化结果。

Accounts 页面配合多平台扩展，已经承接了：

## 8. 运维、发布、同步与质量保障

除了业务功能，工程化和运维链路也改了不少。

### 8.1 `manage.sh` 进程与日志管理

管理脚本已经增强成可直接支撑生产运维：

- 支持 PM2 cluster 多实例
- 支持保留 cluster 模式更新/重启
- 支持直启日志输出开关
- 支持 `rotate-log` 无需重启轮转日志
- 修了 Redis 密码传参等细节问题

### 8.2 上游同步与 fork 发布

又加了一套同步上游与发布辅助能力：

- 自动检查 upstream
- 创建备份分支
- 合并上游更新
- 推送并触发发布
- 发布版本改成以 `VERSION` 文件为准

这部分是为了让这个 fork 能长期演进，而不是一次性改完就停。

### 8.3 模型价格同步

价格同步改造成了更稳的镜像流程：

- 价格文件可以同步到镜像分支
- 生成价格成品文件时附带 hash
- 价格更新失败时仍可回退到本地 fallback 逻辑

### 8.4 覆盖率工作流

这部分最近又继续演进了两层：

第一层是已合入的“变更文件覆盖率”：

- CI 只关心本次改动涉及的源文件
- 输出 Markdown 汇总、LCOV、HTML 报告等产物

第二层是当前工作区新增的“owned lines 覆盖率过滤”：

- 在普通 LCOV 基础上，结合 `git blame` 识别哪些代码行属于当前作者
- 只统计作者本人负责代码行的 line / branch / function 覆盖率
- 生成过滤后的 LCOV、JSON 汇总和 Markdown 汇总
- 把结果追加到 GitHub Actions Summary，并作为 artifact 上传

这个能力的价值在于，后续做变更考核或自查时，可以更清楚地区分“文件整体覆盖率”和“我实际改到的代码行覆盖率”。
